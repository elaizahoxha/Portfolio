Overview

This project focused on building a normalized relational database that could make labor market data easier to explore, compare, and analyze. Instead of treating employment projections as static spreadsheet values, I structured the information into connected entities that support more flexible analytical questions.

The database was designed around occupational data from the U.S. Bureau of Labor Statistics, including employment counts, projected growth, annual openings, median salaries, education requirements, and training requirements. I also extended the system with additional synthetic data to model people, employers, job history, and credentials so the database could support more realistic relationships and richer SQL queries.

The final result was a system that could answer questions about salary, job demand, self-employment, required education, and long-term employment trends in a much more structured way than a flat dataset could.

Project Focus

Relational database design for workforce analytics and occupational trend analysis.

Core Skills

Schema design, normalization, SQL querying, CTEs, ranking, aggregates, analytical functions.

Main Outcome

A queryable system that transforms labor market data into useful insights for career and workforce analysis.

The Problem

Labor market information is valuable, but it is often difficult to work with in its raw form. Many employment datasets are published as spreadsheets or reference tables that are useful for reading but not ideal for deeper analysis. Once someone wants to compare salary against education requirements, or job growth against training barriers, the process becomes more manual and time-consuming.

This project addresses that issue by organizing workforce data into a relational model. Instead of storing everything in one flat structure, the system separates occupations, degrees, training, employers, and people into their own entities, then connects them through bridge tables where needed. That makes the data easier to maintain and much more useful for answering multi-step analytical questions.

The goal was not just to store data, but to design a system that could support more thoughtful questions about the labor market, such as which jobs are growing fastest, which occupations have strong salaries relative to education level, and which careers show higher levels of self-employment.

Data Source

The main source for this project was the U.S. Bureau of Labor Statistics employment projections dataset. That data provided the core occupational metrics used in the database, including 2023 employment, projected 2033 employment, average annual openings, self-employment rate, and median annual wage.

Because the BLS dataset is strong on labor statistics but does not fully model relational activity between people, employers, education, and training, I supplemented it with realistic synthetic data. This made it possible to simulate a fuller system with job histories, credential relationships, employer-provided training, and job requirements.

That combination made the project more useful as a database case study because it demonstrates both structured public data modeling and the ability to create relational depth beyond a single imported table.

Database Architecture

A normalized relational schema designed to reduce redundancy, preserve data integrity, and support more advanced workforce analysis.

Person

Stores demographic and identity-based information for individuals, including name, birthdate, race, and gender. This allows the database to represent people as part of a broader workforce model instead of only storing job statistics.

JobCategory

Serves as the analytical core of the system. This table stores occupation-level labor data such as job title, 2023 employment, 2033 projected employment, self-employment rate, average annual openings, and median salary.

Degree

Stores educational credentials by degree type and field of study. This supports queries about education requirements, salary by degree level, and job-education relationships.

Training

Represents certifications, workshops, and other training programs. This makes it possible to distinguish formal degree requirements from skill-based or on-the-job training pathways.

Employer

Stores organization-level information such as employer name and industry. This supports modeling of who provides training, where jobs may be associated, and how workforce systems connect to employers.

OccupationCategory

Provides a broader category layer for occupations using occupational codes and labels. This adds structure to the system and supports grouping and classification.

Bridge Tables

Tables such as PersonJob, PersonDegree, PersonTraining, JobDegreeRequirement, and JobTrainingRequirement handle many-to-many relationships and make the schema more realistic and scalable.

Integrity Rules

Primary keys, foreign keys, composite keys, and cascade rules were used to maintain consistency across the database and prevent orphaned or mismatched records.

ERD

The entity relationship diagram maps out the structure of the database and shows how core entities connect to one another. A major part of the design was deciding where direct relationships made sense and where bridge tables were necessary to model many-to-many relationships without duplicating data.

Implementation Approach

I implemented the schema using SQL CREATE TABLE statements with explicit primary key and foreign key constraints. The design uses normalized tables so that entities such as degrees, training, and employers are stored once and then linked where needed, instead of repeating the same information in multiple places.

I also used composite primary keys in the bridge tables to represent many-to-many relationships cleanly. For example, one person can have multiple degrees, and one degree type can belong to many people. Similarly, one job may require multiple training programs, and one training program may apply to several jobs.

To support cleaner maintenance, I included cascade rules on several foreign key relationships. That means if a parent record is deleted or updated, the related records stay synchronized rather than becoming disconnected from the rest of the schema.

Key SQL Queries

A big part of this project was not just creating the schema, but using it to answer analytical questions in a way that shows why the structure matters. The queries below highlight some of the most useful patterns in the project, including ranking, grouping, analytical functions, filtering, and relationship-based joins.

1. Top Jobs by Self-Employment

This query ranks occupations by their self-employment rate in 2023. I used a Common Table Expression and the RANK() window function so the results could be ordered from highest to lowest while still being easy to filter afterward.

This is useful because it answers a more targeted career question than a simple list of jobs. Instead of only looking at salary or openings, it highlights occupations where independent work is more common, which could matter for users who are interested in flexibility, entrepreneurship, or contract-based careers.

It also demonstrates a key SQL skill in this project: using window functions to rank results without losing row-level detail. 

WITH RankedSelfEmployed AS (
  SELECT JobID, JobTitle, SelfEmployment2023,
         RANK() OVER (ORDER BY SelfEmployment2023 DESC) AS Rank
  FROM JobCategory
  WHERE SelfEmployment2023 IS NOT NULL
)
SELECT Rank, JobID, JobTitle, SelfEmployment2023
FROM RankedSelfEmployed
WHERE Rank <= 5;

2. Median Salary by Education Level

This query connects job requirements and salary data by joining JobDegreeRequirement, Degree, and JobCategory. It then uses PERCENTILE_CONT(0.5) to calculate the median salary for jobs associated with each degree type.

I chose median instead of average because salary distributions can be skewed by unusually high-paying positions. Median gives a more balanced picture of what compensation looks like within each education category.

This query shows how the database can support questions that combine multiple dimensions of the labor market. It is not just about “what pays well,” but “how does pay relate to the education required to enter that kind of work?” 

SELECT d.DegreeType,
       PERCENTILE_CONT(0.5) WITHIN GROUP (ORDER BY jc.MedianSalary)
       OVER (PARTITION BY d.DegreeType) AS Median_Salary_By_Education
FROM JobDegreeRequirement jdr
JOIN Degree d ON jdr.DegreeID = d.DegreeID
JOIN JobCategory jc ON jdr.JobID = jc.JobID
WHERE jc.MedianSalary IS NOT NULL;

3. Jobs With the Most Average Annual Openings

This query ranks jobs by the number of average annual openings, which helps identify occupations with consistently high labor demand. While projected growth tells one story, openings can reveal a different one because they capture not only new jobs, but also replacement needs and turnover.

This makes the query especially useful for workforce analysis because some roles may not grow at the fastest rate but still generate a large number of openings each year. 

WITH RankedOpenings AS (
  SELECT JobID, JobTitle, OpeningsAvg,
         RANK() OVER (ORDER BY OpeningsAvg DESC) AS Rank
  FROM JobCategory
)
SELECT Rank, JobID, JobTitle, OpeningsAvg
FROM RankedOpenings
WHERE Rank <= 5;

4. Highest Paying Jobs

This query ranks occupations by median salary. It is a straightforward ranking query, but it is still important because it gives a quick way to compare earning potential across roles stored in the system.

In the context of the full database, this query becomes more useful when combined with other dimensions such as education requirements, training requirements, and projected growth. 

WITH RankedSalary AS (
  SELECT JobID, JobTitle, MedianSalary,
         RANK() OVER (ORDER BY MedianSalary DESC) AS Rank
  FROM JobCategory
)
SELECT Rank, JobID, JobTitle, MedianSalary
FROM RankedSalary
WHERE Rank <= 5;

5. Jobs That Require a Bachelor’s Degree

This query filters occupations based on the degree requirement attached to them. By joining jobs to degree requirements, it becomes possible to isolate only those roles that match a specific education pathway.

That kind of query is valuable because education is often one of the first constraints people consider when exploring careers. It can also serve as the foundation for more layered questions, such as which bachelor’s-level roles have the highest salaries or strongest job growth. 

SELECT DISTINCT jc.JobID, jc.JobTitle
FROM JobCategory jc
JOIN JobDegreeRequirement jdr ON jc.JobID = jdr.JobID
JOIN Degree d ON jdr.DegreeID = d.DegreeID
WHERE d.DegreeType = 'Bachelor';

6. Jobs Without Training Requirements

This query uses a LEFT JOIN to find jobs that do not appear in the training requirements table. That makes it useful for identifying occupations with fewer formal barriers to entry.

Ordering those results by salary adds another layer of insight. It shows which jobs are both comparatively accessible and still financially strong, which could be especially useful for workforce advising or career exploration. 

SELECT jc.JobID, jc.JobTitle, jc.MedianSalary
FROM JobCategory jc
LEFT JOIN JobTrainingRequirement jtr ON jc.JobID = jtr.JobID
WHERE jtr.JobID IS NULL
ORDER BY jc.MedianSalary DESC;

7. Top Jobs by Numeric Growth

This query calculates the raw numeric difference between projected 2033 employment and 2023 employment. Instead of percentage growth, it focuses on how many additional positions are expected to exist.

That matters because very large occupations may have a modest percentage increase but still create a huge number of actual jobs. Looking at numeric growth gives a more concrete sense of scale. 

WITH RankedJobsDiff AS (
  SELECT JobID, JobTitle,
         (EmployedCount2033 - EmployedCount2023) AS NewPositionOpenings,
         RANK() OVER (
           ORDER BY (EmployedCount2033 - EmployedCount2023) DESC
         ) AS Rank
  FROM JobCategory
)
SELECT Rank, JobID, JobTitle, NewPositionOpenings
FROM RankedJobsDiff
WHERE Rank <= 5;

8. Jobs With the Lowest Percentage Growth

This query measures change as a percentage instead of a raw count. That gives a different perspective by showing which jobs are projected to grow the least relative to their current size.

Together, this query and the previous one show why multiple measures matter. A job can look very different depending on whether you compare percentage change or total number of positions added. 

WITH RankedJobDiffPercent AS (
  SELECT JobID, JobTitle,
         ROUND(((CAST(EmployedCount2033 - EmployedCount2023 AS FLOAT))
         / EmployedCount2023) * 100.0, 2) AS EmployedCountPercentDiff,
         RANK() OVER (
           ORDER BY ((CAST(EmployedCount2033 - EmployedCount2023 AS FLOAT))
           / EmployedCount2023) * 100.0 ASC
         ) AS Rank
  FROM JobCategory
)
SELECT Rank, JobID, JobTitle, EmployedCountPercentDiff
FROM RankedJobDiffPercent
WHERE Rank <= 5;

9. Average Annual Openings for Management Occupations

This query filters jobs by SOC code prefix and calculates the average annual openings for occupations beginning with 11-, which correspond to management roles in the dataset.

Even though the current join could be simplified, the underlying intent is useful: it shows how occupational codes can be used to analyze subsets of the labor market rather than treating all jobs as one undifferentiated group. 

WITH Soc11 AS (
  SELECT jc.JobID, jc.JobTitle, jc.OpeningsAvg
  FROM JobCategory jc
  WHERE jc.JobID LIKE '11-%'
)
SELECT ROUND(AVG(OpeningsAvg), 2) AS AvgAnnualOpenings_Soc11
FROM Soc11;

10. Employers Offering Multiple Trainings

This query groups employer-training relationships and counts how many distinct training programs each employer provides. It then filters to only show employers with more than one training offering.

This is a good example of how the relational structure supports questions beyond labor statistics alone. It shows how employers can also be analyzed as part of a broader workforce ecosystem. 

WITH EmTrCo AS (
  SELECT et.EmployerID, e.EmployerName,
         COUNT(DISTINCT et.TrainingID) AS DistinctTrainings
  FROM EmployerTraining et
  JOIN Employer e ON et.EmployerID = e.EmployerID
  GROUP BY et.EmployerID, e.EmployerName
)
SELECT EmployerID, EmployerName, DistinctTrainings
FROM EmTrCo
WHERE DistinctTrainings > 1;
